# Persistence contract

`GraphLayoutV2` (schema version 2) is the durable JSON format. It contains graph/composition versions, nodes, links, and explicitly extension-safe metadata. The historical field name `catalogVersion` stores the bound composition version. Links persist a `muted` flag. `GraphLayoutV1` remains accepted and migrates to V2 with links unmuted; unsupported future schema versions produce a structured `schema.future` issue. Selection, camera, hover, history, runtime/session and other transient first-party fields are rejected. Save ordering and object keys are canonicalized.

Known nodes are accepted only at their composition-defined version. Their complete parameter and socket contracts are validated using one declarative `FxNodeValueSchema`: finite bounded numbers (including integer constraints), booleans, strings/enums, finite vectors, bounded colors, and codec-validated JSON widgets. Editable socket `defaultValue` is current persisted state and may differ from the schema's initial `default`; output and non-editable sockets cannot acquire defaults. Every command validates the final document before an atomic commit, so the engine cannot save state its decoder rejects.

Unknown type IDs and future descriptor versions remain opaque, read-only nodes. Valid JSON values—including null, arrays, and objects—round-trip unchanged and render as bounded JSON summaries. Node upgrades are selected only by the node definition's finite declarative migration edges. Each node follows an exact sequential path to its current version; parameter/socket materialization, renames, color-ramp codec conversion, and associated link endpoint rewrites commit transactionally only when the final payload exactly matches the current definition. Missing paths and failed steps retain the original opaque node. Unknown payloads are not recursively interpreted as fxnode session state.

Schema-level graph migration runs before node migration. Node routes then run edge by edge from the payload's `typeVersion`; `catalogVersion` neither chooses nor suppresses a route. A missing first edge, a gap midway through a route, a failed codec or precondition, a destination collision, or a final payload with missing or extra fields makes that node unknown and restores its original durable payload. Earlier successful edges are never exposed. A node already at the current version must match exactly and malformed current data is an error rather than an unknown node. Codec input and output must be finite JSON data; malformed legacy color-ramp stops therefore fail softly instead of escaping the decoder.

Socket renames change both the durable socket key/ID and every matching `fromSocketId` or `toSocketId` for that node. Rewrites are staged until node migrations have resolved, preserve link IDs, order, muted state, and extensions, and never affect links when that node fails. Opposite endpoints may migrate independently. An existing destination socket ID, or a link endpoint for the same node already using that destination ID, is a collision and fails the node transaction. Durable socket shells retain unknown `dataType`/`accepts` values, nullable defaults, metadata, and extensions for unknown nodes; applications must not treat that opaque metadata as a current editable socket contract.

`catalogVersion` does not select migrations; decoded output always uses the bound composition version. Admission rejects hostile or non-JSON object graphs without invoking accessors and enforces fixed depth, size, string, node, link, socket, parameter, issue, and migration-execution budgets. Loading remains atomic and never mutates caller-owned input.

## Live composition and replay

`save()` contains the canonical graph document and the bound `composition.version` under the historical `catalogVersion` name. It does not contain the process-local composition revision, definitions, theme, patch log, undo/redo history, or browser host projection. `catalogVersion` must not be overloaded as a live patch sequence number.

A saved graph can depend on live definitions. Create the editor with the same application identity, version, and resource policies, then reconstruct a compatible composition with `loadComposition()` or deterministic `setTheme`/`setHeaderStyles`/`composeSocket`/`setCompatibility`/`composeNode` calls before invoking `load()`. Replay order matters when definitions reference dynamically composed socket types or migrations depend on earlier authority. Loading command-log save data before reconstructing a compatible composition rejects with `composition.incompatible`.

Removing a definition never deletes its graph instances. Demoted nodes save as ordinary opaque durable payloads and may become known if a compatible definition is introduced later.

## Command-log save data

`getSaveData()` returns an `FxNodeSaveData` envelope with `kind: "fxnode.command-log"`, schema version 2, the complete effective save-time composition, a canonical `GraphLayoutV2` baseline, and the currently applied forward command list. The baseline is necessary to preserve graph identity, metadata, opaque nodes, extensions, and arbitrary initial state that commands alone cannot reconstruct. Undo and redo are not persisted as commands: undo removes the latest applied command, redo restores it, and a new forward command clears that redo branch.

The worker owns the live journal. Calling `getSaveData()` only returns a detached serializable copy and does not increment graph version or emit events. Journal growth is bounded; when a command cannot safely fit the durable replay limits, the worker replaces the journal with a fresh canonical checkpoint of the already committed graph.

`load(saveData)` validates the envelope, save-time composition, baseline, every command, and all aggregate budgets before publishing. The current composition must have the same ID and be an equal or conservative semantic superset: it may add definitions and change presentation or its top-level version, but it cannot remove saved definitions or change defaults, schemas, migrations, socket compatibility, parenting behavior, or other replay semantics. Incompatibility rejects with `composition.incompatible` plus path-specific issues such as `composition.definition-missing`; command-log schema 1 rejects explicitly because it did not contain enough composition information to prove compatibility.

Replay occurs against staged worker state. Failure leaves the live graph, layout, history, journal, projections, and events unchanged. A changed final graph advances the public graph version once and emits one `load` mutation followed by one snapshot. An identical final graph returns `noop` without events or a version increment, while still installing the imported replay history and journal. Existing `load(GraphLayoutV1 | GraphLayoutV2)` remains supported and starts a fresh checkpoint journal.
